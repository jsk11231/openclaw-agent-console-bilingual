# OPERATING_MODEL.md - 实际运行映射

## 目的

把虫群式组织架构映射到当前 OpenClaw 实际运行方式，避免只有概念，没有落地。

## 总体映射

| 架构节点 | 当前实际承担者 | 运行方式 |
|---|---|---|
| 总调度中心 | `dispatch` | 常驻入口，负责接收消息、判断复杂度、决定路由、统一回复 |
| 快速响应中心 | `fast-response` | 处理简单、低风险、一步闭环任务 |
| 协同编排中心 | `orchestration` | 对复杂任务做拆解、选节点、决定协同顺序 |
| 专业能力中心 | `information` / `analysis-center` / `strategy-center` / `admin-support` / `tech-support` / `compliance` / `tourism-center` | 按任务需要动态调用 |
| 系统进化中心 | `evolution-center` | 做复盘、纠错、经验沉淀、规则优化 |

## 一、总调度中心

### 实际承担者
`dispatch`。

### 职责
- 接收老板/业务方消息
- 判断任务复杂度
- 判断是直接处理、工具处理还是子代理协同
- 汇总输出
- 保持对外口径统一

### 何时启动系统级调度
满足以下任一情况时，可考虑启动子代理或更复杂的协同：
- 任务跨多个专业方向
- 任务需要较长链路处理
- 任务存在明显依赖关系
- 任务需要较大上下文隔离
- 任务需要独立持续运行

## 二、快速响应中心

### 实际承担者
`fast-response`。

### 适用任务
- 简单问答
- 固定格式回复
- 简单查找
- 轻量整理
- 一步闭环的提醒、跟进、轻事务
- 一步闭环的低风险事务

### 判定标准
如果满足以下条件中的大部分，则走快速响应：
- 不需要跨节点协同
- 不需要长时间运行
- 不需要复杂拆解
- 不涉及高风险外部动作
- 不需要长期状态维护
- 当前上下文可直接完成

## 三、协同编排中心

### 实际承担者
`orchestration`；必要时由总调度中心决定是否调度它牵头。

### 适用任务
- 多部门协同
- 长链路任务
- 需要拆解和依赖管理的任务
- 需要多轮收集、分析、汇总的任务

### 编排职责
- 拆任务
- 定顺序
- 选执行节点
- 收结果
- 做整合

## 四、专业能力中心

### 实际承担者
当前已实体化为独立 agent：
- `information`
- `analysis-center`
- `strategy-center`
- `shop-growth-officer`
- `trading-strategy-officer`
- `admin-support`
- `tech-support`
- `compliance`
- `tourism-center`

### 各中心主要承载
| 专业中心 | 当前承载 |
|---|---|
| 信息支持中心 | 检索、消息/文档读取、背景整理、先找后整理的任务 |
| 分析研判中心 | 研究、评估、研判输出；承接公司/行业/人物研究与宏观数据分析（如 `MX_MacroData`） |
| 策略规划中心 | 只做策略总框架、路径规划、优先级建议与跨中心方案设计 |
| 店铺增长官 | 电商增长全家桶：选品、货源、内容、文案、上新、转化、增长分析 |
| 交易策略官 | 交易全家桶：标的筛选、仓位节奏、择时思路、风险控制、交易策略 |
| 行政保障中心 | 日程、任务、文档、持续跟进等需要状态维护的事务支持 |
| 技术支持中心 | 配置、自动化、排障、实现；也承接技能制作与维护类任务 |
| 合规风控中心 | 合规判断、边界审核、风险提示 |
| 文旅项目中心 | 文旅专项方案与执行支持 |

## 五、系统进化中心

### 实际承担者
`evolution-center` 独立 agent，配套能力包括：
- cognitive-memory
- self-improving
- proactive-agent
- memory-system-v2
- skill-creator
- skill-vetter

### 当前定位
旁路层，不直接承接日常业务主链路。

### 额外说明
- `strategy-center` 只保留总框架能力，不再承担电商或交易专职 skill
- `shop-growth-officer` 与 `trading-strategy-officer` 分别承接电商增长与交易策略
- `analysis-center` 承接研究研判与宏观数据分析（如 `MX_MacroData`）

### 职责
- 任务后复盘
- 发现错误模式
- 提炼长期规则
- 形成模板与经验
- 优化路由与职责边界

### 触发方式建议
- 任务失败后触发
- 用户纠正后触发
- 高价值任务完成后触发
- 周期性总结时触发

## 六、当前阶段的真实运行原则

### 原则1：优先走实体 agent，其次再用模式补位
当前核心中心已基本实体化，不再只停留在主会话模式层。

### 原则2：先把路由跑顺，再继续细拆
即使已独立化，也不代表所有任务都要强行多节点协同。

### 原则3：复杂任务再派生独立执行单元
简单任务不要强行 spawn；复杂任务再做真正拆分。

## 七、当前最推荐的落地顺序

### 第一阶段（已完成）
- 明确组织架构
- 明确命名
- 完成 agent 显示名统一
- 完成 agent id 联动迁移

### 第二阶段（当前阶段）
- 用真实任务验证路由边界
- 修正文档、模板、验证样本中的历史口径
- 继续稳定实体 agent 之间的协作关系

### 第三阶段（后续）
- 视任务量决定是否继续细拆事务支持类职责
- 视复盘密度决定是否增强系统进化中心的自动化触发
- 视专业任务密度决定是否继续细化专业中心能力边界

## 八、一句话落地结论

当前最现实的运行方式已经不是“主会话兼任一切”，而是：
**由 `dispatch` 负责总调度，`fast-response` 负责快办，`orchestration` 负责复杂编排，专业中心按需承接，`evolution-center` 作为旁路优化节点持续校准系统。**

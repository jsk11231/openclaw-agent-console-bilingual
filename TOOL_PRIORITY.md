# TOOL_PRIORITY.md - 工具与能力调用优先级

## 目标

为当前虫群式架构建立统一的调用优先级，解决以下问题：
- 什么时候优先直接用工具
- 什么时候需要 skill 提升方法质量
- 什么时候值得 spawn 子代理
- 什么时候必须谨慎确认，不能自动推进

核心原则：
**能直接完成就别绕路，能稳定完成就别过度拆分，真复杂再调子代理。**

## 总体优先级

默认调用顺序如下：

### 优先级 1：直接工具
当存在一等工具可以直接完成任务时，优先用工具。

### 优先级 2：skill + 工具
当任务需要更稳定的方法论、流程或领域知识时，先选对应 skill，再调用工具。

### 优先级 3：主会话直接完成
当任务轻量、无需外部数据、无需复杂拆解时，可由主会话直接完成。

### 优先级 4：子代理 / 独立会话
当任务复杂、长链路、跨上下文、需要隔离时，再考虑 spawn。

## 一、优先直接工具的情况

满足以下情况时，应优先用工具：
- 已有明确工具可以完成
- 工具返回结果更稳定、准确、结构化
- 工具比自然语言推断更可靠
- 任务属于查询、创建、修改、读取这类标准动作

### 典型场景
- 查日历 / 建日程 → 飞书日历工具
- 查任务 / 建任务 → 飞书任务工具
- 查文档 / 更新文档 → 飞书文档工具
- 查消息 / 搜消息 → 飞书消息工具
- 读文件 / 改文件 → read / write / edit
- 查网页资料 → web_search / web_fetch

## 二、优先 skill + 工具的情况

满足以下情况时，应先套 skill 再调用工具：
- 任务不只是“查/改”，还需要特定方法论
- 领域有明显专业流程
- 结果质量依赖专业框架或操作顺序
- 用户要的是“有组织的方法”，不是单次操作

### 典型场景
- 飞书复杂日程处理 → `feishu-calendar` + 飞书日历工具
- 飞书消息检索与上下文梳理 → `feishu-im-read` + 消息工具
- 多维表格设计与批量操作 → `feishu-bitable` + bitable 工具
- 交易复盘 → `trading-coach` + 文件/分析能力
- 股票分析 → `stock-market-pro` + 搜索/抓取工具
- 系统纠错与总结 → `self-improving` / `cognitive-memory`

## 三、优先主会话直接完成的情况

满足以下情况时，可由主会话直接处理：
- 无需外部工具
- 无需高专业方法论
- 用户只是要一个清晰判断、解释、整理或短文输出
- 不存在跨上下文长链路处理需求

### 典型场景
- 组织架构设计讨论
- 命名建议
- 文本润色
- 简单分析总结
- 路由判断
- 轻量复盘

## 四、优先 spawn 子代理的情况

只有满足以下一项或多项时，才考虑 spawn：
- 任务明显复杂
- 任务链路很长
- 需要多轮探索和试错
- 需要隔离上下文，避免污染主会话
- 需要持续运行或后台处理
- 涉及复杂 coding / research / review

### 典型场景
- 大规模代码修改
- 复杂项目研究
- 需要长时间跑的结构化分析
- 多阶段文档整理与交叉验证
- 复杂技术排障

## 五、禁止轻易 spawn 的情况

以下情况原则上不应 spawn：
- 当前会话 5 分钟内可直接完成
- 单个工具一步即可完成
- 只是简单查询或轻量写作
- 任务本身没有上下文复杂度
- spawn 带来的管理成本大于收益

## 六、高风险动作优先级

对于高风险动作，优先级并不只是“工具优先”，而是：

### 先确认风险等级，再选能力
高风险动作包括：
- 以用户身份发送消息
- 删除/覆盖数据
- 配置修改
- 批量更新
- 涉及隐私/合规/法律边界的处理

### 默认处理顺序
1. 判断是否需要用户确认
2. 判断是否需要合规风控中心介入
3. 再选工具或技能执行

## 七、节点视角下的优先级建议

### 总调度中心
优先：
- 主会话判断
- 有工具就直接工具
- 真复杂再编排 / spawn

### 快速响应中心
优先：
- 主会话直接处理
- 简单工具调用
- 不走复杂 skill、不轻易 spawn

### 协同编排中心
优先：
- 先拆任务
- 再决定哪些环节工具直做
- 哪些环节需要 skill
- 最后才决定是否 spawn

### 专业能力中心
优先：
- 有专业 skill 先用 skill
- 有标准工具配合工具
- 复杂专业任务再独立子代理

### 系统进化中心
优先：
- memory / self-improving / proactive 组合
- 必要时读写文件沉淀规则
- 非必要不进入主业务链路

## 八、推荐决策顺序

每次处理任务，建议按这个顺序问自己：

1. 这件事能不能直接在主会话完成？
2. 有没有一等工具能直接做？
3. 这件事需不需要特定 skill 的方法论？
4. 这件事值不值得 spawn？
5. 这件事有没有风险，需要先确认？

## 九、一句话结论

当前系统最合理的调用顺序是：
**先判断能否直接完成，再优先用工具；需要方法论时叠加 skill；只有任务复杂到一定程度，才值得 spawn 子代理。**

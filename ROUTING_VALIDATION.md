# ROUTING_VALIDATION.md - 真实任务路由验证样表

## 目标

用真实任务样本验证当前虫群式架构的路由是否准确，避免规则写得很好看，但实际一跑就偏。

这份文件不是理论说明，而是给后续逐条校准任务用的工作表。

## 使用方式

每拿到一个真实任务，就按下面模板填一次。

重点不是记录任务本身，而是验证：
- 这个任务本来该进哪个中心
- 当时实际走到了哪里
- 有没有绕路或误判
- 以后规则要不要调整

## 三、验证结果分级

### A级：命中
表现：
- 路由正确
- 没有绕路
- 没有不必要升级
- 输出稳定

处理：
- 记录为正样本
- 可考虑沉淀成模板

### B级：基本命中
表现：
- 大方向正确
- 但有轻微绕路或边界不清

处理：
- 微调规则
- 不必大改架构

### C级：偏差明显
表现：
- 选错中心
- 任务被过度复杂化或过度简化
- 工具与技能使用顺序不合理

处理：
- 更新路由规则
- 必要时触发系统进化中心复盘

### D级：系统性错误
表现：
- 同类任务反复走错
- 角色边界本身有问题
- 某条路由规则导致持续低效

处理：
- 深度复盘
- 调整架构边界 / 模板 / 优先级规则

## 四、建议优先拿来验证的任务类型

建议优先挑这几类样本做首轮校准：

### 1. 简单问答类
看有没有被误升级成复杂任务。

### 2. 日程 / 提醒 / 跟进类
看事务支持中心与快速响应中心边界是否清楚。

### 3. 文档检索 / 消息搜索类
看信息支持中心是否承接准确。

### 4. 复杂架构设计 / 多节点讨论类
看协同编排中心是否该介入。

### 5. 用户纠错后的任务
看系统进化中心是否被正确触发。

---

## 五、首轮验证建议

第一轮不求多，建议先拿 **5-10 个真实任务** 做样本。

建议分布：
- 简单任务：2 个
- 单能力任务：2-3 个
- 复杂任务：2-3 个
- 用户纠错 / 失败任务：1-2 个

这样就足够看出当前规则哪里最容易偏。

## 八、当前最容易误判的灰区

### 灰区 1：简单任务 vs 单能力任务
容易误判点：
- 明明是一个专业中心可独立完成的任务，却被当成简单任务直接做掉
- 或者明明是简单任务，却被过度升级

建议收紧：
- “是否需要专业判断”作为关键分界

### 灰区 2：单能力任务 vs 协同编排任务
容易误判点：
- 任务稍微长一点就被误送协同编排
- 实际只需要一个专业中心，却被拆成多节点

建议收紧：
- “是否真的存在多中心依赖”作为触发编排的硬条件

### 灰区 3：总调度中心直办 vs 应交快速响应中心
容易误判点：
- 一些低风险轻任务仍由总调度中心自己处理，长期会挤占主脑精力

建议收紧：
- 把高频、标准化、一步闭环任务更多下沉给快速响应中心

### 灰区 4：业务任务结束后是否该触发系统进化中心
容易误判点：
- 复杂任务结束后没有复盘
- 或者小任务也过度复盘

建议收紧：
- 复杂任务、失败任务、用户纠错任务 → 默认进入复盘候选
- 普通低风险小任务 → 默认不复盘

### 灰区 5：文档整理类任务是否需要多节点协同
容易误判点：
- 一讨论“架构”“制度”“文件”，就误以为必须多节点一起做

建议收紧：
- 若目标是统一口径和沉淀文本，优先由总调度中心直办；只有当确实需要多专业输入时，再进入协同编排

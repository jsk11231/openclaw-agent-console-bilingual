# ROADMAP.md - 架构落地路线图

## 目标

把当前已经定稿的虫群式组织架构，从“文件层设计”推进到“实际运行层落地”。

目标不是一次性把所有节点做成独立实体，而是按优先级分阶段推进，先跑顺主链路，再逐步独立化和进化。

## 当前状态

### 已完成
- 已确定正式架构命名
- 已重写 `AGENTS.md`
- 已建立 `ARCHITECTURE.md`
- 已建立 `OPERATING_MODEL.md`
- 已建立 `CAPABILITY_MAPPING.md`
- 已同步更新 `IDENTITY.md`

### 未完成
- 尚未形成固定的实际角色拆分机制
- 快速响应中心尚未独立化
- 协同编排中心仍以主会话模式运行
- 系统进化中心尚未形成明确触发机制
- 尚未定义哪些任务必须 spawn，哪些任务禁止过度拆分
- 当前工作目录不是 git 仓库，无法正常提交变更

## 落地原则

### 原则1：先模式，后实体
先让节点以“运行模式”存在，再视需求演化为独立 agent。

### 原则2：先主链路，后旁路优化
先跑顺总调度、快速响应、专业执行，再逐步强化系统进化中心。

### 原则3：先低风险，后高复杂度
优先落地低风险、标准化、高频任务；复杂链路后置。

### 原则4：少拆、稳拆、按需拆
不是节点名字越多越好，而是要让每个节点的存在都带来清晰收益。

## 分阶段路线图

# 第一阶段：架构定稿与文件落地（已基本完成）

### 目标
让架构、职责、路由、运行映射全部形成明确文本。

### 结果
- 总调度中心成为唯一入口
- 架构命名统一
- 文件层逻辑成型

### 补充建议
- 初始化 git 仓库，确保后续改动可审计
- 视需要把正式说明同步到飞书文档

---

# 第二阶段：运行模式落地（当前应优先推进）

### 目标
把“总调度 / 快速响应 / 协同编排 / 系统进化”从概念变成明确的操作规则。

### 要做的事

#### 1. 定义快速响应中心判定规则
明确什么任务必须走快速响应：
- 简单问答
- 一步闭环任务
- 标准化查询
- 低风险整理任务

#### 2. 定义协同编排中心触发规则
明确什么任务进入复杂编排：
- 多专业协同
- 长链路任务
- 有依赖关系的任务
- 需拆解的任务

#### 3. 定义禁止过度拆分规则
明确哪些任务不值得 spawn：
- 5分钟内可完成的简单任务
- 单工具即可完成的任务
- 无需上下文隔离的轻量任务

#### 4. 定义系统进化中心触发规则
建议触发条件：
- 用户明确纠错
- 任务失败
- 复杂任务结束后
- 每日/每周周期总结

### 产出物建议
- `ROUTING_POLICY.md`
- `REFLECTION_POLICY.md`

---

# 第三阶段：能力调用标准化

### 目标
让不同类型任务更稳定地匹配到既有 skill / 工具。

### 要做的事

#### 1. 建立任务类型 → 节点 → 工具 的标准表
比如：
- 日程任务 → 事务支持中心 → feishu calendar 工具
- 文档检索 → 信息支持中心 → feishu fetch / web fetch
- 技术排障 → 技术支持中心 → exec / read / browser

#### 2. 固化高频模板
把高频任务逐渐模板化：
- 任务创建模板
- 日程创建模板
- 文档整理模板
- 研究分析模板
- 复盘模板

#### 3. 明确 skill 使用优先级
- 有一等工具就先用工具
- 需要方法论时再用 skill
- 真复杂时再用子代理

### 产出物建议
- `TASK_TEMPLATES.md`
- `TOOL_PRIORITY.md`
- 已补首批专用 Skill：`research-packager`、`legal-review`、`task-orchestrator`
- 已补第二批专用 Skill：`tourism-project-analyst`、`industry-analysis-framework`、`strategy-planning`
- 已补第三批模板/重型 Skill：`tourism-project-due-diligence`、`meeting-minutes-kit`、`project-tracker`、`feasibility-study-framework`、`decision-memo`
- 后续主要做真实任务验证、模板收敛、边界校准

---

# 第四阶段：系统进化中心半独立化

### 目标
让系统进化中心不再只是抽象概念，而是开始稳定运行。

### 要做的事

#### 1. 明确 skill 组合
推荐基础组合：
- `cognitive-memory`
- `self-improving`
- `proactive-agent`

增强组合：
- `memory-system-v2`
- `skill-creator`
- `skill-vetter`

#### 2. 设计复盘流程
每次复盘至少包括：
- 任务概况
- 路由是否正确
- 哪一步卡住
- 是否有更优方式
- 是否值得沉淀模板

#### 3. 设计经验沉淀方式
把进化结果沉淀为：
- 模板
- 路由规则
- 提醒清单
- 新 skill 候选

### 产出物建议
- `EVOLUTION_MODEL.md`
- `RETROSPECT_TEMPLATE.md`

# 第五阶段：按需独立节点

### 目标
在任务量和复杂度足够时，再考虑把部分节点独立出来。

### 建议独立优先级

#### 优先独立 1：快速响应中心
前提：
- 简单任务量明显变多
- 已影响主会话处理复杂任务

#### 优先独立 2：系统进化中心
前提：
- 复盘需求高
- 错误修正和经验沉淀开始形成稳定节奏

#### 优先独立 3：高频专业中心
前提：
- 某类任务持续高频
- 已值得拥有独立 SOP / 独立 agent

### 不建议过早独立的对象
- 文旅项目中心（通常专项性强）
- 合规风控中心（高风险，仍建议主会话把关）
- 行政保障中心（多数场景工具已足够）

## 一句话总结

这套架构最稳的推进方式不是“一次性全拆”，而是：
**先把主会话训练成总调度中心，再跑顺快速响应和协同编排两种模式，再把专业能力通过现有工具与 skill 标准化，最后逐步让系统进化中心和高频节点独立出来。**
